Accessibility

Presentation of accessibility for our site.

This page covers how we made our site accessible for all users by making coding choices for universal design.

Structure

General layout, clear headers and high-resolution images. The pages has wide margins for Understandable: Implicitly imply the content without needing to read everything. Keeping headers in correct order and using semantic tags.

Readability

We chose the font Verdana, with sans-serif as a backup. The justification for this choice was based on it's legibility and accessible design by being simple and unembellished. The font should clearly differentiate the letters well even in small text sizes. We've avoided fonts featuring “imposter letter shapes”. These include characters that are visually similar, but not the same, for example uppercase and lowercase Is, lowercase Ls, and the number 1 (Iil1). We've ensured that the font size is in a relative unit, like rem and percentages, to scale it properly based on individual user needs. This is because using absolute units, like pixels, limits the option to increase or decrease text size without zooming in on everything else on the page. All paragraphs use 1.3em line-height and letter-spacing is set to 0.04rem to make the text easier to read.

Colors

We implemented both light and dark modes with high contrast to enhance readability. Links are adjusted to be clearly visible in both modes while maintaining association with traditional hyperlink colors.

Choices and justification

Lightmode and darkmode = contrast on both! darkmode links are lighter than lightmode links while still being assosiated to hyperlinks

Images

We've made sure to include a descriptive alternative text on every image for visually impaired users, but also in case any images should fail to load. !Images AND icons should have this! PNG, WebP, JPEG Backup image formatting nescessary? Multiple versions of one image to support different screen sizes

Interactivity

Screen readers (aria-label). Tabindex for non-mouse users

Lighthouse Testing

We started testing the accessibility of our site by using the automated Lighthouse tests in Google Chrome. The first test gave a score of 22/25 and highlighted missing labels in forms and problematic lists. The main source of this was the hamburger menu items starting with anchor-tags instead of list-tags and missing labels for the input type checkbox that is used as a click receiver.

The second test gave an overall score of 25/25 after we rearranged the menu item tags and added labels to our code. This prompted us to dig deeper and manually check for code improvement opportunities.

Later in the semester, another Lighthouse test was done while the hamburger menu was opened. This third test gave a score of 24/25 based on ARIA attributes not being used as specified for the elements role.
